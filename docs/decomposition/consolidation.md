# proxima consolidation / RISC-dedup discipline log

Sibling to `discipline.md` (which tracked the original 2026-05 *split* into
flat crates). This log tracks the *reverse*: RISC-dedup (P1 reuse-first) —
collapse duplicated erasure/combinator code to the primitives, and consolidate
the crate explosion (peaked at 96) to primitives + thin domain-function crates.
Driver: "no new primitives — we already have them; most crates should be a few
domain functions over the primitives."

## Gate (every fold row)

A row is not sealed (✅) until, on the rebased-onto-`main` branch:
`cargo check --workspace` clean AND `cargo nextest run` green for every touched
crate (+ the umbrella). Landed = pushed to `main` (fast-forward/rebase, no
feature branch on origin). Behaviour-preserving: existing behaviour is the
oracle (P14) — a fold that changes a public runtime behaviour is a bug, not a
fold. No blank cells; deviations documented in the row.

## Held (naming-gated / design-heavy — NOT executed unattended)

  (`SendPipe<Request<Bytes>,Response<Bytes>,ProximaError>` + name/background_tasks)
  is a 2nd thing called `Pipe` and must be renamed. Proposed vocabulary awaiting
  ruling: **Handler** (the served pipe), **Handle** (erased ref), **Listener**
  (speaks the wire), **App** (mounts + runs); **no `Service`** (tower's Service =
  handler AND app). NOT http-specific — HTTP is one protocol among pgwire/redis/
  mqtt/… A `Handler` IS a pipe → stays in `proxima-pipe`. Sub-question: keep the
  Handler surface in `proxima-pipe` or move to the serve crate.
- **J** rename served trait (~238 impl sites) — gated on the naming ruling.
- **C** `proxima-http` (fold h1/h2/h3/codecs→1) + pgwire/redis→protocols. Fold
  approved; served-trait relocation naming-gated. Watch for the same h1/h2↔listen
  cycle constraint hit in B*.

  → shared-fn pattern (`retry.rs` is the one correct exemplar), THEN a

  Also generalize `retry_exec::Retry`→`Replayable`; share circuit-breaker rules
